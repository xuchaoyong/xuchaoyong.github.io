Lua的变量

1. 没有预定义类型

Lua是一种动态类型的语言，变量没有预先定义类型，所以变量可以代表任何一个类型。

print(type(a))  --> nil ('a' is not initialized)
a = 10
print(type(a))  --> number
a = "a string"
print(type(a))  --> string
a = print
print(type(a))  --> function

a这个变量可以指向任何的对象类型。

2. 局部变量

Lua中的局部变量要用local关键字来显示定义，不用local显示定义的变量就是全局变量：

g_var = 1         -- global var
local l_var = 2   -- local var

2.1作用域

局部变量的生命周期是有限的，它的作用域仅限于声明它的块（block）。 一个块是一个控制结构的执行体、或者是一个函数的执行体再或者是一个程序块（chunk）。我们可以通过下面这个例子来理解一下局部变量作用域的问题：

x = 10
local i = 1         --程序块中的局部变量i

while i <=x do
  local x = i * 2   --while循环体中的局部变量x
  print(x)          --打印2, 4, 6, 8, ...(实际输出格式不是这样的，这里只是表示输出结果)
  i = i + 1
end

if i > 20 then
  local x           --then中的局部变量x
  x = 20
  print(x + 2)      --如果i > 20 将会打印22，此处的x是局部变量
else
  print(x)          --打印10， 这里x是全局变量
end

print(x)            --打印10

2.2 局部变量的好处

使用局部变量的一个好处是，局部变量可以避免将一些无用的名称引入全局环境，避免全局环境的污染。另外，访问局部变量比访问全局变量更快。同时，由于局部变量出了作用域之后生命周期结束，这样可以被垃圾回收器及时释放。

在Lua中，应该尽量让定义变量的语句靠近使用变量的语句，这也可以被看做是一种良好的编程风格。在C这样的语言中，强制程序员在一个块（或一个过程）的起始处声明所有的局部变量，所以有些程序员认为在一个块的中间使用声明语句是一种不良好地习惯。实际上，在需要时才声明变量并且赋予有意义的初值，这样可以提高代码的可读性。对于程序员而言，相比在块中的任意位置顺手声明自己需要的变量，和必须跳到块的起始处声明，大家应该能掂量哪种做法比较方便了吧？

3. 内存回收

在C/C++中，定义一个在变量就会分配一段内存(栈或者堆)，在栈上分配的内存会随着变量作用域结束而自动释放，在堆上分配的内存一般需要手动释放。

在Lua中，系统为对象分配内存，而不为变量分配内存。

local a  --> 系统不会分配内存给a
local a = {}  --> 系统为一个空表分配内存

上面例子中，系统会为空表{}分配内存，变量a只是对表的一个引用。

i = 100
if i > 20 then
  local x = {}
  b = x
else
  print(x) 
end

print(x)  --> nil

以上例子中，x是一个块局部变量，在else处作用域失效，那么x指向的空表内存会被释放吗？

答案是：不会。因为全局变量b还指向空表，即空表还存在一个引用。

3.1 Lua简单回收算法

Lua最简单的垃圾回收方法是引用计数法，每个分配内存的对象都有一个引用计数。如上例中，空表的引用计数为2，到达else时，引用计数为1。如果一个对象的引用计数为0，那么这个对象占用的内存自动被释放。所以，在上例中空表的内存没有被释放，因为全局变量b指向空表，空表对象的引用计数为1。

然后引用计数回收算法有个很明显的缺点：循环引用。如果对象a引用对象b，对象b同时引用对象a，使用引用计数法不能顺利的回收内存，将存在内存泄露。

> a = {}
> b = {}
> a['other'] = b
> b['other'] = a   -- now we have a cyclic reference
> a,b = nil,nil    -- oh dear, we don't have a reference to the 2 tables now
>                  -- there are two tables which are unreachable

如上例中最后a和b指向nil，但是两个空表都不会被释放，因为循环引用的原因。

3.2 标记擦除

为了克服上述循环引用的问题，Lua使用标记擦除法进行垃圾回收。首先记录所有的变量记录他们引用的对象，如果发现变量对某个对象不可达，那么就对对象进行内存回收。如循环引用例子中，两个空表都不可访问，那么两个空表对象的内存被回收。

这种方法可以克服引用计数法的问题，但是缺点是需要记录所有变量，实时性差。

4. 变量本质

在Lua中变量的本质是引用(地址)，指向对象地址的引用。

比较两个变量是否相等时，只有他们指向的对象地址相等时，它们才是相等的。

a = {}; a.x = 1; a.y = 0
b = {}; b.x = 1; b.y = 0
c = a

a == c
a ~= b

如上例中，只有两个变量引用同一个对象时，它们才是相等的。

require and module in lua

Posted on 2015-09-03   |  

1. require模块发生了什么

在我们的V8资源侧Lua代码中通常将模块使用module封装起来，使用的时候通过require来引用module，然后利用模块名访问module里面的定义变量和方法。那么，在这个require过程中到底发生了什么？在module声明的时候为什么一般要使用package.seeall参数？如下面例子所示

#文件名test.lua
module("test", package.seeall)

class = {}
local localcalss = {}

function good()
  print("good")
end

function class:good()
  print("class good")
end

通过require模块，调用定义的方法。

#文件名example.lua
require "test"

local var = test.good()
local var1 = test.class.good()

require函数的实现机制有些复杂，这里给出源代码。

function require (name)
    if not package.loaded[name] then -- module not loaded yet?
        local loader = findloader(name)
        if loader == nil then
            error("unable to load module " .. name)
        end
        package.loaded[name] = true -- mark module as loaded
        local res = loader(name) -- initialize module
        if res ~= nil then
            package.loaded[name] = res
        end
    end
    return package.loaded[name]
end

3. module做了什么

为什么在定义module的时候要有参数package.seeall？如果去掉这个参数，会发生什么？

#文件名test.lua
module("test")

class = {}
local localcalss = {}

function good()
  print("good")
end

function class:good()
  print("class good")
end

执行example.lua，输出

lua: ./test.lua:4: attempt to call global 'print' (a nil value)
stack traceback:
        ./test.lua:4: in function 'good'
        example.lua:3: in main chunk
        [C]: in ?

为什么系统函数print在module的定义中不包括package.seeall参数就会报错呢？

这是由Lua脚本的执行环境引起的，默认情况下Lua脚本的执行环境为全局表，即_G。在开头打印_G全局表，可以看到print是_G表中的一个元素，类型为函数。在全局表的执行环境下，直接使用print函数没有问题，Lua执行时会在_G表中查找print这个元素，能够找到。

但是在module中，如果定义时不包括package.seeall参数，那么Lua的执行环境是私有的环境(programing in Lua Chapter14)，其实模块test的定义可以展开module部分。

#文件名test.lua
#module("test") 等价于

local modname = "test"
local M = {}
_G[modname] = M
package.loaded[modname] = M

setfenv(1, M)

class = {}
local localcalss = {}

function good()
  print("good")
end

function class:good()
  print("class good")
end

通过setfenv函数把当前的执行环境切换为M这个表，而在M中没有定义print函数，所以出现错误。

增加package.seeall参数等同于以下代码。

#文件名test.lua
#module("test") 等价于

local modname = "test"
local M = {}
_G[modname] = M
package.loaded[modname] = M

#设置M表的元表为_G
setmetatable(M, {__index = _G})

setfenv(1, M)

class = {}
local localcalss = {}

function good()
  print("good")
end

function class:good()
  print("class good")
end

通过设置M表的元表为_G，当执行print查不到的时候，看是否定义了元表的__index元素，然后在__index指向的表中找到print的方法。

4. 官方不建议使用module关键字

1. package.seeall 这种方式破坏了模块的高内聚，原本引入”test”模块只想调用它的good()函数，但是它却可以读写全局属性，例如 “test.os”

例如，在example.lua中这么写

#文件名：example.lua
require "test"

local var = test.good()
local var1 = test.class.good()

test.io.write("test io write")

输出：

test io write

这样显然是违背module的本意的。

2. module函数压栈操作引发的副作用，污染了全局环境变量。例如module(“test”) 会创建一个test的table，并将这个table注入全局环境变量中，这样使得没有引用它的文件也能调用test模块的方法。

例如，private模块并没有require “test”这个模块，但是它里面却可以通过全局变量表来访问test模块里面的方法。

#文件名private.lua
module "private"

_G.test.good()

输出：

good

在private.lua没有引入test这个module的时候确仍然可以使用全局变量进行访问，这也是不当的。

总结：虽然有以上的问题，但是当前代码仍然采用module的形式。

open function Trap in C++

Posted on 2015-07-24   |  

1. Background

Give the following code which comes from unixdomain/mycat.c in UNP.

#include <stdio.h>
#include <iostream>
#include <stdlib.h>
#include <sys/socket.h>
#include <errno.h>
#include "unixsock.h"

static int write_fd(int fd, void *ptr, int nbytes, int sendfd);

int main(int argc, char **argv)
{
    int fd;
    char fdstr[20];
    int value = 0;
    if (argc != 4)
        ERR_RETURN_VALUE("usage: openfile <sockfd> <filename> <mode>");

    if ((fd = open(argv[2], atoi(argv[3])) < 0))
    {
        ERR_RETURN_NOVALUE("file non-exist");
        exit((errno > 0) ? errno : 255);
    }

    if (write_fd(atoi(argv[1]), const_cast<char *>(""), 1, fd) < 0)
    {
        ERR_RETURN_NOVALUE("write_fd failed");
        exit((errno > 0) ? errno : 255); 
    }

    close(fd);

}

Can you find the bug in the above code?
The bug costs me more than 4 hours to figure it out.

2. Output

The purpose of above code is to transfer a file descriptor from the child process to parent process. Then the parent program will read content the file descriptor.

However, I find that my program could not correctly output the content of the file.

Then I try to write something to the file in the child process and I can read the content in the parent process, which totally confused me.

In the beginning, I think there must be something wrong with the opening mode of the file. Therefore, I used fcntl function to check and change the mode of the descriptor. However, it does not make a little chance.

However, during the procedure, I find a strange matter that I get different mode by using fcntl in two differnt program to get the status of mode.

OK.

I find I just made a so stupid mistake.

3. Solution

#include <stdio.h>
#include <iostream>
#include <stdlib.h>
#include <sys/socket.h>
#include <errno.h>
#include "unixsock.h"

static int write_fd(int fd, void *ptr, int nbytes, int sendfd);

int main(int argc, char **argv)
{
    int fd;
    char fdstr[20];
    int value = 0;
    if (argc != 4)
        ERR_RETURN_VALUE("usage: openfile <sockfd> <filename> <mode>");

    /*The open function bracket*/
    if ((fd = open(argv[2], atoi(argv[3]))) < 0)
    {
        ERR_RETURN_NOVALUE("file non-exist");
        exit((errno > 0) ? errno : 255);
    }

    if (write_fd(atoi(argv[1]), const_cast<char *>(""), 1, fd) < 0)
    {
        ERR_RETURN_NOVALUE("write_fd failed");
        exit((errno > 0) ? errno : 255); 
    }

    close(fd);

}

Wrong Code: (fd = open(argv[2], atoi(argv[3])) < 0)

fd = 0, which refers to STDIN. Because open will return 1 which is bigger than zero, fd = 1 < 0 = false = 0.

Right Code: ((fd = open(argv[2], atoi(argv[3]))) < 0)
